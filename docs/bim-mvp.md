# BIM-MVP: текущее техническое состояние

## 1. Назначение текущего BIM-MVP

Текущий BIM-MVP в проекте "Технадзор Онлайн" решает первую практическую задачу:

- загрузка IFC-файла
- извлечение базовых BIM-элементов
- запись элементов в `projects/{projectId}/elements`
- использование этих элементов в модулях геодезии и геометрии

На этом этапе BIM не включает:

- 3D-viewer
- clash detection
- полную BIM-навигацию по модели
- сложную Autodesk-интеграцию
- автозаполнение всех возможных параметров конструкции

## 2. Как работает IFC-import

### 2.1. Общая цепочка

1. Пользователь выбирает IFC-файл в блоке `BIM / IFC`.
2. Клиент отправляет `POST /bim/import-ifc`.
3. Сервер принимает файл как сырой `application/octet-stream`.
4. Серверный parser разбирает IFC, фильтрует поддерживаемые сущности и нормализует поля.
5. Перед записью сервер очищает ранее импортированные элементы с тем же `sourceModelId`.
6. Новые элементы записываются в `projects/{projectId}/elements`.

### 2.2. Поддерживаемые IFC-типы

На текущем этапе импортируются:

- `IfcSlab` -> `slab`
- `IfcColumn` -> `column`
- `IfcWall`
- `IfcWallStandardCase` -> `wall`
- `IfcBeam` -> `beam`
- `IfcStair`
- `IfcStairFlight` -> `stair`

### 2.3. Серверные точки

- импорт IFC: `POST /bim/import-ifc`
- удаление импортированных BIM-элементов: `DELETE /bim/elements`

Обе точки работают только после проверки доступа к проекту на сервере.

## 3. Структура `projects/{projectId}/elements`

Каждый документ в `projects/{projectId}/elements` описывает один импортированный BIM-элемент.

### 3.1. Базовые поля

- `elementId`: внутренний идентификатор элемента в проекте
- `sourceModelId`: идентификатор импорта / модели
- `ifcGuid`: IFC GUID элемента
- `type`: нормализованный BIM-тип (`slab`, `column`, `wall`, `beam`, `stair`)
- `floor`: этаж, если parser смог надёжно определить `IfcBuildingStorey`
- `axes`: нормализованные оси, если parser смог их надёжно определить
- `mark`: марка / обозначение элемента
- `projectX`: проектная координата X
- `projectY`: проектная координата Y
- `projectH`: проектная отметка H
- `importedAt`: серверное время записи элемента

### 3.2. Дополнительные геометрические поля

Сейчас в `elements` также могут записываться размеры, если parser смог извлечь их надёжно:

- `thickness`
- `length`
- `width`
- `height`
- `sectionWidth`
- `sectionHeight`

### 3.3. Какие размеры реально используются сейчас

- `slab`: в первую очередь `thickness`
- `beam`: `width`, `height`
- `column`: `sectionWidth`, `sectionHeight`

`length` уже может сохраняться в `elements`, но пока не является основой автоподстановки в UI.

## 4. Какие BIM-поля уже используются в геодезии

В геодезии BIM уже используется как источник выбора элемента и проектных координат.

### 4.1. Что можно выбрать

Пользователь выбирает BIM-элемент из `projects/{projectId}/elements` с поиском и фильтрами:

- поиск по BIM-элементам
- фильтр по типу
- фильтр по осям

### 4.2. Какие поля уже подставляются

После выбора BIM-элемента геодезия может автоподставить:

- `type` -> тип конструкции
- `projectX`
- `projectY`
- `projectH`
- `mark`
- `axes`

### 4.3. Как это сохраняется

В геодезическом узле сохраняется BIM-привязка:

- `bimElementId`
- `bimSourceModelId`
- `bimIfcGuid`
- `bimType`
- `bimMark`
- `bimAxes`
- `bimProjectX`
- `bimProjectY`
- `bimProjectH`

Если элемент позже недоступен в `elements`, сама геодезическая запись не удаляется, а UI показывает, что BIM-источник недоступен.

## 5. Какие BIM-поля уже используются в геометрии

В геометрии BIM уже используется как источник выбора элемента, контекстной BIM-summary и части проектных размеров.

### 5.1. Что подставляется стабильно

После выбора BIM-элемента геометрия может подставить:

- `type`
- `mark`
- `axes`
- `floor`, если он реально есть

### 5.2. Какие числовые размеры уже подключены

На текущем этапе geometry-BIM использует только надёжные размеры:

- `slab -> thickness`
- `beam -> width`, `height`
- `column -> sectionWidth`, `sectionHeight`

### 5.3. Что пока не подставляется автоматически

Пока остаются ручными:

- большинство размеров стен
- параметры лестниц
- сложные размерные характеристики, если parser не смог извлечь их уверенно
- размеры, которые либо не хранятся в `elements`, либо пока не подключены в UI

## 6. Ограничения текущего parser

### 6.1. `floor`

Этаж извлекается через `IfcBuildingStorey` и связанные spatial/placement-цепочки. Ограничения:

- если в IFC нет `IfcBuildingStorey`, `floor` останется `null`
- parser не подставляет случайные этажи
- для проверки качества `floor` нужен IFC, где storey действительно есть

### 6.2. `axes`

Оси уже нормализуются и пригодны для фильтрации, но остаются ограничения:

- не все IFC содержат хорошие данные по осям
- часть элементов остаётся без `axes`
- если оси нельзя извлечь надёжно, parser оставляет `null`, а не мусор

### 6.3. Размеры

Часть размеров уже извлекается надёжно, но coverage неполный:

- не у всех `beam` есть надёжные `width/height`
- не у всех `column` есть надёжные `sectionWidth/sectionHeight`
- если размер нельзя получить уверенно, он остаётся `null`

Это ожидаемое поведение текущего safe-MVP: отсутствие размера лучше, чем сомнительное значение.

## 7. Как работает удаление BIM-элементов

Удаление BIM-элементов выполняется отдельным действием в UI через подтверждение.

### 7.1. Базовый сценарий

1. Пользователь нажимает `Очистить IFC`.
2. UI показывает подтверждение, так как действие опасное.
3. Клиент вызывает `DELETE /bim/elements`.
4. Сервер удаляет BIM-элементы из `projects/{projectId}/elements`.

### 7.2. Как определяется область удаления

- если известен `sourceModelId`, удаляются только элементы текущего импорта
- если `sourceModelId` неизвестен, возможно удаление всех `elements` проекта

### 7.3. Что происходит после удаления

- очищается client-side состояние текущего IFC
- обновляется BIM-статус в UI
- пересобираются BIM-списки в геодезии и геометрии
- существующие `geoNodes` и `geomChecks` не удаляются
- если старая BIM-связка осталась в сохранённой проверке, UI показывает, что источник BIM больше недоступен

## 8. Текущее состояние BIM-MVP по модулям

### 8.1. Уже работает

- IFC-import
- запись элементов в Firestore
- удаление импортированных BIM-элементов
- выбор BIM-элемента в геодезии
- выбор BIM-элемента в геометрии
- BIM-summary и визуальное отделение BIM-данных от ручного ввода
- частичная автоподстановка проектных размеров в геометрии

### 8.2. Пока не входит в текущий этап

- viewer
- интеграция BIM в остальные модули кроме геодезии и геометрии
- полное извлечение этажей для всех IFC
- полное извлечение размеров для всех типов элементов

## 9. Практический вывод

Текущий BIM-MVP уже пригоден для рабочих сценариев:

- загрузить IFC
- получить `elements` в Firestore
- выбрать BIM-элемент в геодезии
- выбрать BIM-элемент в геометрии
- использовать BIM как источник типа, марки, осей, проектных координат и части проектных размеров

Следующий логичный этап развития:

- расширять parser/import для более широкого покрытия `floor`
- расширять coverage надёжных размеров в `elements`
- аккуратно подключать новые BIM-данные в geometry/UI только там, где значения действительно извлечены надёжно
